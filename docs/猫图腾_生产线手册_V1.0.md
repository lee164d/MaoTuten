=====================================
第1章｜项目定义
=====================================

项目代号：猫图腾
赛道：8-15秒短视频
主发布平台：抖音
目标受众：一二线城市18-28岁年轻女性

一句话定义：用猫作为目标受众的高配替身人格，交付高审美、可传播、可系列化的情绪代餐内容。

精致疯 = 高级感容器 + 可共情的离谱内核。

项目不是什么：不是宠物号，不是泛搞笑号，不是空镜美学号，不是为追热点而稀释调性的号。

=====================================
第2章｜角色设计
=====================================

2.1 主角

一只银白色极简漫改猫。

内核：毫不费力的不在乎。不是冷酷（冷酷是刻意的），不是傲慢（傲慢需要观众），不是可爱（可爱是讨好的）。是"我就这样，你爱看不看"。

目标受众想代入的不是一只好看的猫，是一种状态——做荒诞的事时的完全淡定。

视觉签名（不可变的识别锚点）：
- 半睁眼：不屑于完全睁开，视觉签名
- 微歪头：审视式歪头，不是可爱式歪头
- 银白底色 + 淡紫灰耳尖/尾尖
- 娇小纤细体型
- 极简线条，无多余装饰
- 表情幅度极小：喜怒哀乐的变化约等于零

辨识度来源：不是装饰（铃铛/腮红/星星=套路），是性格感。看到它的第一反应是"这猫怎么回事"而不是"好可爱"。

2.2 情绪反转系统

猫核（上述视觉签名）是常量，服装/配饰/场景配色是变量。
同一只猫通过视觉标记的增减实现情绪切换：
- 素颜态：无配饰，半睁眼偏向"懒得动"，用于M1铺垫
- 全装态：加配饰+眼妆变化+配色变化，半睁眼偏向"就这？"，用于M2/M3反转

反转信号不靠面部微表情（视频生成工具做不到），靠：
- 运动状态突变（走→停）
- 镜头语言硬切（景别突变）
- 服装/配饰的出现或消失
- 场景配色的冷暖切换

2.3 视觉壳轮换

猫核不变，视觉壳按当期梗的原生语言轮换：
- 武侠/古风（雪山救狐狸等古装梗）
- KPOP（韩流相关梗）
- 日式/中式/其他文化框架

视觉壳通过场景美术和服化道实现，不是让猫表演该文化的动作。

=====================================
第3章｜内容方法论
=====================================

3.1 核心原则

情绪价值是唯一交付物。梗是入场券，不是骨架。
梗关联极轻——受众0.1秒联想到该梗的10%就够。
短视频的密度指标：情绪密度 > 信息密度 > 情节密度。

3.2 选梗流程

1. B站梗指南最新一期，按时间倒序逐个过
2. 问：这个梗最能满足什么情绪价值
3. 能满足→猫IP能否交付（任意视角和角色组合）
4. 不能直接满足→能否变体满足
5. 都不行→下一个
6. 通过→找参考源头→出作战卡

3.3 四筛子

情绪命中｜她会不会停下来
高级感｜冷、利、克制、有设计感
猫适配｜任意视角和组合能不能承载
暗扣可设计性｜有没有一个不说出口的讨论缝隙

3.4 叙事结构

M1（铺垫）→ M2（转折）→ M3（高潮）
转折信号必须是硬的（运动突变/镜头硬切/视觉标记变化），不靠微表情。
暗扣点必须内嵌在内容中，禁止显式互动引导。

3.5 参考源头规则

没有参考源头不写作战卡。
没有视觉基准不设计角色。
参考源头的作用是给视觉节奏和第一秒钩子设计提供锚点。
参考源头由人类亲自观看视频确认——"第一秒能不能抓住我"的判断无法委托。

=====================================
第4章｜生产线
=====================================

4.1 协作架构

当前阶段为两方协作：

人类（决策者+操作者）
- 拥有最终决策权和全局直觉
- 负责所有关键节点确认和不可委托的工具操作
- 唯一能看到"全局画面"的角色

AI策划者（Claude对话）
- 拥有项目上下文记忆
- 负责写prompt、判断生成结果（技术层面）、给修正方向
- 不能操作浏览器或工具
- 不了解工具最新能力，不确定时先问再写
- 美感判断归属人类，策划者只做技术判断（一致性/构图/prompt遵从度）

中间层AI（Chrome侧边栏）为可选辅助工具，不是信息流必经节点。
当其图片上传能力或共享存储方案跑通后，可重新评估三方架构。

4.2 工具链

关键帧生成：MJ（niji 7，插画风格）
关键帧精修/分镜生成：Gemini Nano Banana（同一对话链内编辑）
视频合成（首选）：可灵3.0 Omni（分镜模式，多图@引用）
视频合成（备选）：即梦Seedance 2.0（首尾帧模式）
视频合成（备选）：SORA（单图+prompt模式，无图片分镜能力）
后期合成：CapCut Web

4.3 生产阶段

阶段0 · 选梗
→ 从B站梗指南最新期逐个过
→ AI策划者批量评估，给排序
→ 人类选定

阶段1 · 找参考源头
→ AI策划者定义采集方向
→ 人类亲自看视频选定1-3个参考
→ AI策划者拆解参考源头（钩子/节奏/情绪弧线）

阶段2 · 作战卡
→ AI策划者基于梗+参考源头写作战卡
→ 人类确认

阶段3 · 角色+关键帧
→ MJ出角色概念稿（1张，单体全身，干净背景）
→ 人类选片
→ 人类上传到Gemini，AI策划者写分镜prompt
→ Gemini同一对话链内逐帧生成
→ AI策划者判断一致性，人类判断美感
→ 迭代直到全套关键帧通过

阶段4 · 视频生成
→ AI策划者写视频prompt（适配目标工具格式）
→ 人类选择工具并提交生成
→ AI策划者审核（技术层面），人类审核（动态体感）
→ 迭代直到通过

阶段5 · 后期
→ AI策划者给出字幕方案
→ 人类在CapCut执行
→ AI策划者终审（技术），人类终审（整体）

阶段6 · 发布+复盘
→ AI策划者给发布文案和标签
→ 人类发布
→ 24-48小时后回收数据
→ AI策划者复盘

4.4 人类关键决策节点（6个）

0. 选定梗
1. 选定参考源头（必须亲自看视频）
2. 确认作战卡
3. 确认角色+关键帧（美感判断）
4. 确认视频素材（动态体感判断）
5. 确认成片

=====================================
第5章｜已验证结论
=====================================

以下结论来自两条视频的实际生产，不是推测。

5.1 工具能力事实

MJ niji 7：
- 插画风格猫出图稳定
- prompt写"character design sheet"会出多角度散布图而非4个独立方向
- prompt写"realistic/photorealistic"与niji 7矛盾
- --cref跨帧一致性不如Gemini同对话链

Gemini Nano Banana：
- 同一对话链内角色一致性可靠（已验证：姿态变化、视角变化、场景变化、风格推进）
- 新对话链=新角色=一致性崩坏
- 跨工具（GPT生成的图 vs MJ生成的图）一致性不可靠

可灵3.0 Omni：
- 分镜模式有效：多图@引用+逐镜头独立时长/描述/参考图绑定
- 10秒4镜头叙事弧线完整
- 角色一致性全程保持
- 720p关闭音画同步=6灵感值/秒
- 景别接近时过渡好，景别+视角同时大跳时有轻微问题

即梦Seedance 2.0：
- 无原生多镜头分镜面板
- @引用是全局的不能镜头级绑定
- 带参考图约40积分/5s，纯文字约25积分——积分变化可验证参考图是否挂上
- 中文prompt原生优化

SORA：
- Storyboard每个Scene只有文字输入，没有图片上传
- 普通模式可上传1张图作为首帧
- 角色一致性纯靠文字控制，不可靠

5.2 流程事实

- 没有爆款参考源头就写作战卡=凭空造概念
- 没有视觉基准就设计角色=闭门造车
- 文档越厚产出越少=过度文档化
- 微表情prompt（瞳孔变化/微微低头）视频工具不可靠
- 转折信号靠运动状态突变+镜头语言硬切
- MJ出母本→Gemini链式编辑=正确路径，跳过Gemini直接MJ --cref=跳层
- 换工具=换角色——一致性只在同一工具同一会话内成立
- 美感判断由人类做，策划者做技术判断——两者不可混淆

5.3 角色设计事实

- 写实风AI猫容易掉入恐怖谷或塑料感
- 抖音/小红书上"好看的漫改猫"已严重同质化，大眼+腮红+装饰物=套路
- 辨识度来自内核外化，不是视觉堆砌
- 巫师猫的爆火验证了"风格化/超现实AI猫"比写实猫更有爆款潜力
- 宠物经济猫消费市场1520亿元，增速5.2%，90后+00后宠主占比69%——猫是确定的增长主题

=====================================
第6章｜禁止项
=====================================

内容层面：
- 低俗/吵/乱/土
- 显式互动引导（评论区告诉我/你遇到过吗/点赞关注）
- 旁白解说
- 真黑暗内容
- 为适配梗牺牲情绪密度
- 为蹭热点偏离账号垂直定位

角色层面：
- 猫穿完整人类服装（可加配饰，不可全身拟人穿衣）
- 猫做不像猫的动作（猫味是硬约束）
- 画面中猫不可识别

生产层面：
- 没有参考源头就写作战卡
- 没有视觉基准就设计角色
- 跳过Gemini直接用MJ --cref生后续帧
- prompt中写视频生成工具做不到的微妙状态
- 假设工具能力存在但未验证
- 把第三方推测数据当官方事实写入文档

=====================================
第7章｜框架管理规则
=====================================

7.1 本文档的性质

这是生产线的操作系统，不是项目的全部知识。
具体的prompt模板、工具参数格式、作战卡模板作为附录单独维护。
角色概念稿、已确认的关键帧等视觉资产存储在项目文件夹中，不写入本文档。

7.2 更新规则

每完成3条视频的完整生产+发布+数据回收后，评估一次是否需要更新本文档。
两次评估之间，框架冻结。发现的问题记录在"待评估清单"中，不即时修改框架。
例外：框架的核心假设被实战证伪时（如三方架构的信息流假设），允许即时修正。

7.3 待评估清单（当前）

- 中间层AI的图片上传能力是否已解锁
- 共享存储方案（Google Drive文件夹）是否可替代截图搬运
- 可灵Omni vs 即梦SD2的成本效率对比（需要更多样本）
- 角色V3在不同梗类型下的适配度（温情向梗是否需要调整）
- AI漫剧200亿市场对猫图腾的商业模式启示
